# Canonical Travel Schema (CTS)
# Version 1.0 — April 2026
# Status: Internal / Pre-Publication Draft
# Maintained by: EMOSS Consulting / TOS Platform

---

## Abstract

The Canonical Travel Schema (CTS) is a unified, type-safe data schema
for representing travel products across all verticals — hotels,
experiences, transfers, flights, rail, and packages — in a single
discriminated union format.

CTS eliminates the need for per-vertical data models by defining a
shared base object with type-specific extensions. Any travel product,
regardless of supplier or vertical, maps to one CTS object. Any
system that speaks CTS can consume products from any supplier without
vertical-specific parsing logic.

CTS is designed to be:
- **Vertical-agnostic** — one schema covers all travel product types
- **Supplier-agnostic** — supplier-specific fields are namespaced extensions
- **Agent-ready** — structured for consumption by AI agents and LLMs
- **Extensible** — platforms and suppliers can extend without breaking compatibility
- **Interoperable** — unknown extensions must be preserved, never dropped

## 1. Design Principles

### 1.1 Discriminated Union

Every CTS object has a `type` field that identifies its vertical.
This is the discriminator — all type-specific logic branches on this
field. The base fields are identical regardless of type.

```json
{ "type": "HOTEL", ... }
{ "type": "EXPERIENCE", ... }
{ "type": "TRANSFER", ... }
{ "type": "FLIGHT", ... }
{ "type": "RAIL", ... }
{ "type": "PACKAGE", ... }
```

### 1.2 Shared Base

All CTS objects share the same base fields regardless of type.
A system that only understands the base can process any CTS object —
it may not understand all extension fields, but it can display
the title, price, location, and availability of any product.

### 1.3 Type Extensions

Each type has a corresponding namespace (`cts:hotel`, `cts:experience`,
etc.) containing fields that only apply to that vertical. Type
extensions are part of the published standard.

### 1.4 Supplier Extensions

Suppliers may add proprietary fields using the `x-{supplier_slug}`
namespace. These fields are not part of the standard and are not
normalized. They are preserved for booking and cancellation calls
that require supplier-native identifiers.

### 1.5 Platform Extensions

Platforms implementing CTS may add intelligence fields using the
`x-{platform}` namespace. These are never part of the published
standard. They represent platform-specific metadata such as dedup
scores, sync provenance, or pricing anomaly flags.

### 1.6 Preservation Rule

**A CTS implementation MUST preserve all fields it does not
understand.** Unknown `x-` fields must pass through every pipeline
stage unchanged. This is the foundational rule that enables
extensibility without fragmentation.

## 2. Core Fields

The following fields are defined for all CTS types. All implementations
MUST support these fields.

### 2.1 Required Core Fields

| Field | Type | Description |
|-------|------|-------------|
| `option_id` | UUID string | Unique identifier for this option. Generated by the producing system. |
| `type` | enum | Vertical discriminator. See Section 3. |
| `title` | string | Human-readable product name. Always `title` — never `hotel_name`, `tour_name`, etc. |
| `supplier_slug` | string | Identifier of the supplier that produced this record. |
| `supplier_raw_ref` | string | Opaque supplier reference. Required for re-price and booking calls. Must be preserved exactly. |
| `price` | CTSPrice | Pricing block. See Section 2.3. |
| `availability` | CTSAvailability | Availability block. See Section 2.4. |

### 2.2 Optional Core Fields

| Field | Type | Description |
|-------|------|-------------|
| `description` | string | Full product description. |
| `origin` | CTSLocation | Origin or primary location. See Section 2.5. |
| `destination` | CTSLocation | Destination location (point-to-point products). |
| `images` | string[] | Array of image URLs. |
| `policies` | CTSPolicies | Cancellation, change, and baggage policies. See Section 2.6. |
| `depart_utc` | ISO8601 | Departure datetime in UTC. FLIGHT / RAIL / TRANSFER. |
| `arrive_utc` | ISO8601 | Arrival datetime in UTC. FLIGHT / RAIL / TRANSFER. |
| `checkin_date` | YYYY-MM-DD | Check-in date. HOTEL. |
| `checkout_date` | YYYY-MM-DD | Check-out date. HOTEL. |

### 2.3 CTSPrice

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `amount_usd` | number | Yes | Normalized price in USD. |
| `original_amount` | number | Yes | Price in supplier's native currency. |
| `original_currency` | string | Yes | ISO 4217 currency code. |
| `fx_rate` | number | Yes | Exchange rate applied: original_amount × fx_rate = amount_usd. |
| `net_amount_usd` | number | No | Wholesale/net price for B2B contexts. |
| `markup_applied` | boolean | No | True if markup has been applied to amount_usd. |

### 2.4 CTSAvailability

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `status` | enum | Yes | See status values below. |
| `seats` | integer | No | Available seat count. EXPERIENCE / FLIGHT. |
| `rooms` | integer | No | Available room count. HOTEL. |
| `max_passengers` | integer | No | Vehicle capacity. TRANSFER. |
| `hold_expiry` | ISO8601 | No | Time at which this availability quote expires. |

**Availability status values:**

| Value | Meaning |
|-------|---------|
| `CONFIRMED` | Available and bookable. |
| `LOW_AVAILABILITY` | Available but limited — fewer than 5 units remaining. |
| `SOLD_OUT` | No availability. |
| `CANCELLATION_FEE_APPLIES` | Available but free cancellation window has passed. |
| `PRICING_TYPE_UNCERTAIN` | Available but pricing model unclear (B2B vs retail ambiguity). |
| `DURATION_UNKNOWN` | Available but duration could not be determined. |

### 2.5 CTSLocation

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `type` | enum | Yes | AIRPORT / HOTEL / COORDINATES / CITY |
| `city` | string | Yes | City name. |
| `country` | string | Yes | ISO 3166-1 alpha-2 country code. |
| `timezone` | string | Yes | IANA timezone identifier e.g. Europe/Paris. |
| `iata_code` | string | No | IATA airport or city code. |
| `latitude` | number | No | WGS84 latitude. |
| `longitude` | number | No | WGS84 longitude. |

### 2.6 CTSPolicies

| Field | Type | Description |
|-------|------|-------------|
| `cancellation` | CTSCancellationPolicy | See below. |
| `change` | object | Change/amendment policy. Structure TBD in v1.1. |
| `baggage` | object | Baggage allowance. FLIGHT only. Structure TBD in v1.1. |

**CTSCancellationPolicy:**

| Field | Type | Description |
|-------|------|-------------|
| `free_until` | ISO8601 | Datetime until which cancellation is free. |
| `penalty_schedule` | array | Array of `{ hours_before: number, charge_pct: number }`. |
| `policy_source` | enum | `SUPPLIER` / `DEFAULT_APPLIED` / `CONFLICT_RESOLVED_RESTRICTIVE` |

## 4. Type Extensions

Type extensions are published as part of the CTS standard. They are
included in a CTS object only when the corresponding `type` value
is set. All type extension fields are optional unless marked required.

### 4.1 cts:hotel

Applies when `type = HOTEL`.

| Field | Type | Description |
|-------|------|-------------|
| `star_rating` | number | Star rating (0.0–5.0). |
| `meal_plan` | enum | `RO` (room only) / `BB` (bed & breakfast) / `HB` (half board) / `FB` (full board) / `AI` (all inclusive). |
| `room_type` | string | Room category name from supplier. |
| `room_code` | string | Supplier room code. |
| `check_in_time` | HH:MM | Standard check-in time. |
| `check_out_time` | HH:MM | Standard check-out time. |
| `amenities` | string[] | List of amenity codes or names. |
| `room_count` | integer | Number of rooms in this rate option. |

### 4.2 cts:experience

Applies when `type = EXPERIENCE`.

| Field | Type | Description |
|-------|------|-------------|
| `duration_minutes` | integer | Duration in minutes. |
| `experience_category` | enum | See category values below. |
| `meeting_point` | string | Pickup or meeting point description. |
| `languages` | string[] | Available languages (ISO 639-1 codes). |
| `min_participants` | integer | Minimum group size. |
| `max_participants` | integer | Maximum group size. |
| `included` | string[] | What is included in the price. |
| `excluded` | string[] | What is not included. |
| `operator_name` | string | Name of the tour operator. |

**Experience category values:**

`CULTURE` / `SPORT` / `FOOD` / `NATURE` / `ADVENTURE` /
`WELLNESS` / `ENTERTAINMENT` / `TRANSPORT` / `NIGHTLIFE` /
`SHOPPING` / `FAMILY` / `OTHER`

### 4.3 cts:transfer

Applies when `type = TRANSFER`.

| Field | Type | Description |
|-------|------|-------------|
| `transfer_type` | enum | `SHARED_TRANSFER` / `PRIVATE_TRANSFER` / `LUXURY_TRANSFER` |
| `vehicle_class` | enum | `SEDAN` / `VAN` / `MINIBUS` / `BUS` / `LIMOUSINE` |
| `max_passengers` | integer | Maximum passenger capacity. |
| `max_luggage` | integer | Maximum luggage items. |
| `pickup_type` | enum | `MEET_AND_GREET` / `CURBSIDE` |
| `trip_id` | UUID | Links outbound and return transfer legs. |
| `inbound_flight` | string | Flight number for airport pickup coordination. |
| `return_trip_id` | UUID | option_id of the return leg. |
| `passenger_manifest_required` | boolean | Whether passenger names are required at booking. |

### 4.4 cts:flight (Planned — v1.1)

Applies when `type = FLIGHT`. Fields TBD. Will include:
`segments`, `itinerary`, `fare_class`, `fare_basis`,
`baggage_allowance`, `cabin_class`, `carrier_code`,
`flight_number`, `operating_carrier`.

### 4.5 cts:rail (Planned — v1.1)

Applies when `type = RAIL`. Fields TBD. Will include:
`segments`, `train_number`, `service_class`, `operator`,
`station_origin`, `station_destination`.

### 4.6 cts:package (Planned — v1.2)

Applies when `type = PACKAGE`. Will reference an array of
constituent CTS objects. Structure TBD.

## 5. Supplier Extensions

Supplier extensions use the `x-{supplier_slug}` namespace. They are
defined by the supplier or by the platform implementing the supplier
connector. They are not part of the CTS standard.

**Rules:**
- Namespace MUST be `x-` followed by a lowercase, hyphen-separated
  supplier identifier matching the supplier's `supplier_slug`
- Fields inside the namespace are unstructured — the CTS standard
  makes no requirements on their names or types
- Supplier extensions MUST be preserved through all pipeline stages
- Supplier extensions MUST NOT be used for display or booking logic
  by systems that do not explicitly understand that supplier

**Example — HotelBeds:**
```json
{
  "x-hotelbeds": {
    "rateKey": "opaque~rate~key~token",
    "boardCode": "BB",
    "net": 280.00,
    "allotment": 5,
    "paymentType": "AT_WEB",
    "rateClass": "NOR"
  }
}
```

**Example — Bridgify:**
```json
{
  "x-bridgify": {
    "experienceId": "BRG-12345",
    "operatorCode": "OP-789",
    "cancellationClass": "C",
    "voucherType": "ELECTRONIC"
  }
}
```

## 6. Platform Extensions

Platform extensions use the `x-{platform}` namespace. They represent
metadata added by the platform during processing — intelligence fields,
provenance, quality signals. They are never part of the published
standard.

**Rules:**
- Namespace MUST be `x-` followed by the platform identifier
- Platform extensions MUST be stripped before exposing CTS objects
  to external consumers unless the consumer explicitly requests them
- Platform extensions MUST NOT affect the semantic meaning of
  core or type extension fields

**Example — TOS Platform:**
```json
{
  "x-tos": {
    "dedup_score": 0.82,
    "dedup_decision": "UNCERTAIN",
    "candidate_pair_id": "uuid",
    "inventory_source": "STATIC_CACHE",
    "last_synced_at": "2026-04-13T06:00:00Z",
    "sync_job_id": "uuid",
    "pricing_anomaly": false,
    "prompt_triggered": "integration.hotelbeds.rate_key_expiry_risk",
    "media_quality": "HIGH"
  }
}
```

## 8. Versioning

### 8.1 CTS Core

Versioned with semver. The version is declared in a `cts_version`
field on every object (optional in v1.0, required from v1.1).

| Change Type | Version Increment | Example |
|-------------|------------------|---------|
| Breaking change to core fields | Major (1.0 → 2.0) | Renaming `title` to `name` |
| New optional core field | Minor (1.0 → 1.1) | Adding `tags[]` to base |
| Bug fix / clarification | Patch (1.0 → 1.0.1) | Fixing timezone format spec |

### 8.2 Type Extensions

Each type extension (`cts:hotel`, `cts:experience`, etc.) versions
independently from the core. Extension version is declared in
`cts:{type}_version` (optional in v1.0).

### 8.3 Supplier and Platform Extensions

`x-` extensions are unversioned by definition. They are
proprietary namespaces and may change without notice. Systems
that consume `x-` fields are responsible for handling changes.

### 8.4 Compatibility

A CTS v1.x implementation is backward compatible with all v1.x
objects. A v2.0 implementation must explicitly declare support
for v1.x objects if backward compatibility is required.

## 9. Conformance

A conformant CTS implementation MUST:

1. Support all required Core fields (Section 2.1)
2. Preserve all optional Core fields it receives, even if it
   does not use them
3. Preserve all `x-` extension fields it receives unchanged
4. Not modify `supplier_raw_ref` under any circumstances
5. Not use `x-{supplier}` fields for display or booking logic
   unless explicitly implementing that supplier connector
6. Strip `x-{platform}` fields before exposing objects to
   external consumers (unless the consumer requests them)

A conformant CTS implementation SHOULD:

1. Support the Type Extension for its primary vertical
2. Validate objects against the Core schema before emitting them
3. Set `policy_source` when applying default cancellation policies
4. Generate a fresh `option_id` for every object it produces

A conformant CTS implementation MAY:

1. Add its own `x-{platform}` extension fields
2. Support multiple type extensions
3. Implement extensions not yet in the published standard,
   provided they use the `x-` namespace

The following types are planned for future CTS versions. Field

## 11. Relationship to Existing Standards

CTS is not a replacement for existing travel standards. It is a
normalization layer that sits above them.

| Standard | Scope | CTS Relationship |
|----------|-------|-----------------|
| NDC / IATA | Flight distribution | CTS `FLIGHT` type maps from NDC responses |
| OTA / OpenTravel | Hotels, cars, activities | CTS maps from OTA-format responses |
| HTNG | Hotel technology | CTS maps from HTNG property data |
| GBFS | Bike/scooter sharing | Future — possible TRANSPORT type |
| Schema.org | General web data | CTS `origin` aligns with schema.org/Place |
| GeoJSON | Geospatial data | CTS location fields are GeoJSON-compatible |

CTS does not require suppliers to implement any standard. It
normalizes whatever format the supplier uses into a single
canonical shape. The mapping from supplier format to CTS is
the responsibility of the supplier connector — not the supplier.
